기본 정책

01. 목적

  • 프로젝트의 퍼블리싱 품질과 유지보수 기준을 통일한다.
  • 프로젝트 간 코드 일관성과 재사용성을 확보한다.
  • 프로젝트별 산출물 구조, 마크업, 스타일, 접근성 기준을 문서화한다.
  • 신규 인력이 동일한 기준으로 빠르게 프로젝트에 적응할 수 있도록 한다.

02. 적용 범위

  • 본 가이드는 HTML 기반 프로젝트를 기준으로 하며 다음 항목에 적용한다.
  • HTML, CSS(SCSS), JavaScript(modal, tab, accordion 등의 UI 인터랙션)
  • 이미지 및 리소스 관리
  • 접근성
  • 산출물 관리
  • 프로젝트 특성에 따라 일부 규칙은 변경될 수 있다.

03. 기본 원칙

  • 유지보수가 쉬운 구조로 작성한다.
  • 재사용 가능한 컴포넌트를 우선으로 한다.
  • 웹 접근성을 기본으로 고려한다.
  • 동일한 프로젝트에서는 동일한 규칙을 유지한다.

규칙

01. 크로스 브라우징

  • 기본 렌더링 검수는 Chromium 기반 Chrome에서 진행한다.
  • 브라우저 검수 우선순위는 Chrome → Naver Whale → Microsoft Edge 순서로 적용한다.
  • Chrome, Whale, Edge는 최신 2개 버전을 기준으로 한다.
  • Safari (macOS / iOS 최신 2개 버전)
  • Samsung Internet (Android)
  • Chrome 검수가 불가능하면 Whale, Edge 순으로 확인하며 정적 파일 검사로 브라우저 검수를 대체하지 않는다.
  • Firefox는 프로젝트 요구사항에 따라 지원한다.
  • 프로젝트 요구사항에 따라 Microsoft Edge의 IE Mode를 통한 호환성을 지원할 수 있다.
  • 프로젝트 특성에 따라 지원 브라우저가 변경될 수 있다.

02. 운영 스크립트 로드 순서

  • 외부 라이브러리, 기능별 플러그인, 공통 UI 스크립트 순서로 의존성을 고려해 불러온다.
  • Swiper 등 기능별 라이브러리는 사용하는 페이지에서만 로드하고, 사용하지 않는 플러그인은 불러오지 않는다.
  • 운영 스크립트는 body 종료 직전에 선언하며, 가이드 전용 guide-vendor.js와 guide.js는 운영 ui.js와 구분해 관리한다. 
<script src="../assets/js/vendor/jquery-4.0.0.min.js"></script>
<!-- Swiper 사용 페이지에서만 로드 -->
<script src="../assets/js/vendor/swiper-bundle.min.js"></script>
<script src="../assets/js/ui.js"></script>

03. Git 작업 메시지 규칙

  • [파트명] 년도월일 - 직관적인 작업 내용 작성
  • 단, Commit에는 하나의 작업만 포함하는 것을 원칙으로 한다.
  • ex) [pub] yymmdd - 버튼 간격 조정
  • ex) [pub] yymmdd - 모달 팝업 작업
  • ex) [pub] yymmdd - #25234 카드형 컴포넌트 수정

04. Git 작업 관리

  • 작업 시작 전 현재 브랜치와 변경 파일을 확인하고 원격 저장소의 최신 내용을 반영한다.
  • 브랜치 생성과 병합 방식은 각 프로젝트에서 정한 Git 운영 정책을 우선 적용한다.
  • 한 커밋에는 동일한 목적의 변경 사항만 포함하고 관련 없는 파일을 함께 커밋하지 않는다.
  • 커밋 전 변경 파일과 diff를 확인하여 누락된 파일이나 의도하지 않은 변경이 없는지 점검한다.
  • 공통 CSS와 JavaScript 변경 시 영향을 받는 가이드와 프로젝트 화면을 확인한 후 커밋한다.
  • 생성 파일, 개인 설정 파일, 로그와 불필요한 임시 파일은 저장소에 포함하지 않는다.
  • 충돌이 발생하면 대상 파일의 기존 구조와 양쪽 변경 목적을 확인한 후 필요한 코드만 반영한다.
  • 공유 브랜치의 이력을 임의로 변경하는 강제 push는 프로젝트 관리자의 확인 없이 사용하지 않는다.

05. 컴포넌트 작성 기준

  • 가이드 페이지명과 클래스명은 역할을 바로 이해할 수 있는 단순하고 간결한 단어 조합으로 작성한다.
  • 새 컴포넌트를 만들기 전에 form, button, tab, status, overlay, display, image의 기존 스타일을 먼저 확인한다.
  • 컴포넌트 페이지에는 실제 예제, 복사 가능한 pre 예제, 사용 조건과 접근성 주의사항을 함께 작성한다.
  • 공통 규칙을 수정하거나 신규 항목을 추가할 때는 퍼블리싱 가이드와 프로젝트 가이드의 관련 HTML, CSS, pre, JavaScript를 함께 반영한다.
  • HTML 마크업과 CSS는 한 줄로 압축하지 않고 요소, 선택자, 속성 단위로 줄을 나눠 작성한다.
  • 동작이 필요한 경우에만 JavaScript를 추가하고 aria 상태와 실제 화면 상태를 동일하게 변경한다.
  • 신규 영역에는 주석을 작성하고 components.css import와 산출물 리스트 이력을 함께 갱신한다.

필수 구조와 조합 기준

컴포넌트별 필수 구조와 조합 방법
컴포넌트 필수 구조 조합 팁
button 기본 클래스에 유형·크기 클래스를 추가한다. 배열은 group 클래스에 지정하고 버튼 자체의 유형 클래스와 분리한다.
form label의 for와 입력 요소의 id를 연결한다. 안내·오류 문구는 aria-describedby로 연결하고 group 안에서 순서를 유지한다.
checkbox·radio·toggle input 다음에 연결된 label을 배치한다. 상태 스타일이 input과 label의 연결에 의존하므로 두 요소 사이에 다른 요소를 넣지 않는다.
tab·accordion 제어 버튼과 콘텐츠의 id·ARIA 속성을 연결한다. 콘텐츠 안에는 다른 컴포넌트를 넣을 수 있지만 제어 버튼과 패널 구조는 변경하지 않는다.
tooltip 트리거와 설명을 같은 tooltip 컨테이너에 둔다. 기본 컨테이너 구조를 유지하고 필요한 위치 클래스만 추가한다.
modal·bottom sheet dialog 안에서 header·body·footer 역할을 구분한다. 버튼 배열은 row·stack 유형 클래스로 정하고 레이어 안에 다른 레이어를 중첩하지 않는다.
Swiper swiper, swiper-wrapper, swiper-slide 구조를 유지한다. 기본 구조에 navigation과 pagination 요소를 필요한 유형만 추가한다.
  • 필수 구조는 접근성 상태 연결과 JavaScript 동작을 위한 최소 범위이며, 콘텐츠 영역에는 필요한 컴포넌트를 조합할 수 있다.

06. 구현 및 렌더링 검수

  1. 1차: 기존 마크업 구조, CSS 규칙, 클래스 네이밍과 컴포넌트를 유지해 구현한다.
  2. 2차: 구현 완료 후 파일 내용만 확인하지 않고 실제 브라우저 렌더링을 기준으로 검수한다.

텍스트와 코드 긴 제목, 버튼 문구, 파일명과 pre 코드 예제의 줄바꿈 및 콘텐츠 높이 변화를 확인한다.
메뉴와 콘텐츠 기존 LNB·GNB와 콘텐츠 영역의 겹침, 고정 요소의 가림 및 스크롤 위치를 확인한다.
공통 영향 변경한 CSS 파일을 참조하는 전체 가이드 페이지에서 높이 변화와 반응형 충돌 여부를 확인한다.
  • 렌더링 문제 발견 시 즉시 수정하지 않고 원인, 발생 해상도, 영향받는 페이지와 수정 범위를 먼저 공유한다.
  • 정적 파일 검사와 JavaScript 문법 검사는 렌더링 검수를 대체할 수 없다.

프로젝트 구조

01. 운영 구조

  • 루트 guide는 다음 프로젝트의 기준이 되는 Publishing Guide Master이다.
  • project는 실제 작업 폴더이며 폴더 자체를 고객에게 전달한다.
  • project/guide는 프로젝트 산출물의 CSS를 직접 불러와 실제 화면과 동일한 스타일을 사용한다.
  • 프로젝트 UI 스타일은 project/assets에서만 수정하고 project/guide에 복사하지 않는다.
  • project/guide/assets/css/import.css의 프로젝트 CSS 경로를 변경하거나 컴포넌트 CSS를 중복 추가하지 않는다.

02. 산출물 리스트 관리

  • 전체 페이지의 작업 범위와 진행 상태를 한 화면에서 공유하기 위해 작성한다.
  • 기획·디자인·개발에서 동일 화면을 찾을 수 있도록 Depth, 페이지명과 경로를 관리하며, 사용하지 않는 화면 ID는 하이픈으로 표시한다.
  • 상태와 최종 수정일을 갱신하고, 비고에는 신규·수정·참조 이력을 최신순으로 남긴다.
  • 페이지 추가, 경로 변경, 기능 수정 또는 검수 완료 시 산출물 리스트를 함께 수정한다.
  • pub-pagelist.html에서 산출물 현황과 작성 방법을 확인한다.

├─ index.html 가이드 메인
├─ pub-pagelist.html 퍼블리셔 전용 가이드 산출물 리스트
├─ html/ 규칙 및 컴포넌트 가이드 HTML

├─ css/ 기본·컴포넌트·가이드 스타일
├─ js/ 가이드 실행 스크립트
├─ guide.js LNB와 가이드 예제 동작 전체
├─ vendor/ 직접 수정하지 않는 외부 코드
├─ jquery-4.0.0.min.js 가이드 공통 jQuery
├─ swiper-bundle.min.js 슬라이드 예제에서만 불러오는 Swiper
└─ guide-vendor.js 코드 하이라이트·스와이프용 가이드 외부 플러그인
└─ fonts/ 가이드에서 사용하는 로컬 웹폰트
  • 외부 라이브러리는 js/vendor 한 곳에서만 관리하고 기능별 하위 폴더는 추가하지 않는다.
  • jQuery는 가이드 공통으로 불러오고 Swiper는 슬라이드가 있는 페이지에서만 불러온다. guide-vendor.js는 코드 하이라이트·스와이프 등 가이드 전용 외부 플러그인이다.
  • 직접 작성하는 가이드 기능은 guide.js에만 작성하고 vendor 파일은 수정하지 않는다.
  • fonts와 images는 vendor의 하위 항목이 아니라 assets의 리소스 유형이며, 실제 사용하는 경우에만 추가한다.

  • HTML에는 import.css 하나만 연결하고 import.css에서 외부 라이브러리, style.css, guide.css 순서로 불러온다.
  • guide.css에는 가이드 화면 전용 스타일만 작성하며 프로젝트 컴포넌트 스타일을 중복 작성하지 않는다.
  • 새 컴포넌트 CSS는 components 폴더에 추가하고 components.css에서 한 번만 import한다.
  • 컴포넌트는 기본 클래스에 유형·크기·배열·상태 클래스를 조합하며, 부모 요소와 자식 순서에 불필요하게 의존하지 않는다.

01. 마크업

  • HTML은 의미를 기준으로 작성한다.
  • 의미를 표현할 수 있는 시멘틱 태그를 우선 사용하고, 적절한 태그가 없는 경우 div를 사용한다.

<!DOCTYPE html>

<head>
    <meta charset="UTF-8">

  • JS는 렌더링 성능을 위해 body 종료 직전에 선언한다.

성능 최적화

01. CLS 대응 기준

  • 이미지 크기와 Lazy Loading 구현은 이미지 가이드 기준을 따르고, width, height 또는 aspect-ratio로 렌더링 전 표시 영역이 확보되는지 확인한다.
  • 비동기 콘텐츠는 Skeleton UI 또는 Placeholder를 제공하고 실제 콘텐츠와 유사한 크기를 유지한다.
  • 초기 렌더링 이후 기존 콘텐츠를 밀어내는 방식으로 새 요소를 삽입하지 않는다.
  • 애니메이션은 transform과 opacity를 우선 사용하고 width, height, top, left 변경은 지양한다.
  • display: none에서 block으로 전환되는 요소는 초기 표시 상태와 전환 이후의 레이아웃 영향을 함께 확인한다.
  • min-height 고정은 실제 원인을 해결할 수 없는 경우에만 마지막 수단으로 사용한다.

컴포넌트 작성 규칙

  • 컴포넌트의 초기 상태와 JavaScript 적용 후 상태가 동일한 표시 영역을 유지하도록 작성한다.
  • Swiper, Carousel, Tab, Accordion은 초기화 전에도 컨테이너와 필요한 콘텐츠 영역을 확보한다.
  • JavaScript 초기화가 레이아웃 이동의 원인인 경우 CSS만으로 임시 대응하지 않고 초기화 흐름과 상태를 먼저 확인한다.
  • 로딩, 빈 상태, 오류 상태는 콘텐츠 영역의 높이가 급격히 달라지지 않도록 함께 설계한다.

마크업 체크리스트

  • 이미지에 width와 height 속성을 선언했는지 확인한다.
  • 동영상과 iframe에 크기 또는 비율을 확보하는 구조를 적용했는지 확인한다.
  • Lazy Loading 대상이 로드 전에도 필요한 표시 공간을 유지하는지 확인한다.
  • 초기 렌더링 이후 DOM 삽입으로 기존 콘텐츠가 이동하지 않는지 확인한다.

02. LCP 대응 기준

  • 첫 화면의 주요 콘텐츠와 LCP 후보 요소는 초기 HTML에 포함하고 불필요한 JavaScript 완료를 기다려 표시하지 않도록 한다.
  • LCP 이미지에는 loading="lazy"를 적용하지 않으며 실제 표시 크기에 맞는 이미지 리소스를 사용한다.
  • LCP 이미지가 명확한 경우에만 preload와 fetchpriority="high" 적용을 검토한다.
  • 첫 화면 렌더링을 지연시키는 CSS, 폰트, JavaScript와 외부 리소스의 필요성을 확인한다.

03. 이미지 최적화

  • 콘텐츠 특성에 맞게 WebP, AVIF 등 적절한 형식과 압축 품질을 사용한다.
  • 화면 크기별로 필요한 이미지 크기가 다른 경우 picture, source 또는 srcset 사용을 검토한다.
  • 첫 화면 밖 이미지는 loading="lazy" 적용을 검토하되, LCP 후보 이미지에는 적용하지 않는다.

04. 웹폰트 최적화

  • 폰트 파일 등록과 형식 선택은 CSS 작성 규칙의 폰트 기준을 따른다.
  • font-display 설정은 폰트 표시 지연과 대체 폰트 적용 후 레이아웃 이동을 함께 고려해 결정한다.
  • 첫 화면에 반드시 필요한 폰트만 preload 적용을 검토하며, 불필요한 preload는 추가하지 않는다.
  • 웹폰트 적용 전후 줄바꿈과 요소 높이 변화가 없는지 주요 화면에서 확인한다.

05. 최종 성능 체크리스트

  • Chrome DevTools Performance에서 Layout Shift와 Affected Node, Culprit을 확인한다.
  • 수정 전후 CLS 발생 원인과 수정 이유를 함께 기록한다.
  • 레이아웃 이동이 발생한 경우 CSS만으로 해결 가능한지 판단하고, JavaScript 초기화와 동적 동작의 영향을 함께 확인한다.
  • LCP 후보 요소, 렌더링 차단 리소스, 이미지와 웹폰트 로딩 순서를 확인한다.
  • 주요 반응형 해상도에서 초기 로딩, 이미지 로딩, 비동기 콘텐츠 노출 후 레이아웃 이동을 확인한다.
  • Google Search Console의 CLS 수치만으로 원인을 단정하지 않고 실제 Performance 기록을 함께 확인한다.

웹 접근성

01. 기본 원칙

  • 웹 접근성은 모든 사용자가 웹 콘텐츠에 쉽게 접근할 수 있도록 보장하는 것을 의미한다.
  • 접근성 기준을 인식하고 준수해야 한다.

02. Skip Navigation 

<!-- Skip Navigation -->
<div id="skip-nav">
    <a href="#contents">본문 바로가기</a>
</div>

<header class="header">
    헤더
</header>

<main id="contents" class="container">
    <div class="contents">
        메인
    </div>
</main>

<footer class="footer">
    푸터
</footer>
  • 페이지 최상단에 배치한다.
  • href는 본문(main) 영역의 id와 연결한다.
  • 기본적으로 숨김 처리하고 :focus 상태에서만 노출한다.
  • 키보드 포커스로 접근 가능해야 한다.

03. 이미지(image) 

<img src="logo.png" alt="회사명">
<img src="bg.png" alt="">
  • 이미지에는 대체 텍스트를 제공한다.
  • 장식용 이미지는 alt 속성을 비워둔다.

04. 제목(Heading) 

<h1>페이지 제목</h1>
<h2>섹션 제목</h2>
<h3>하위 제목</h3>
  • h1 다음에 바로 h4처럼 heading 단계를 건너뛰지 않는다.

05. 버튼(button) 

<a href="/notice/noti-list.html">공지사항 보기</a>
<button type="button">팝업 열기</button>
  • 페이지 이동은 a, 기능 실행은 button을 사용한다.

06. 폼(form) 

<label for="user-name">이름</label>
<input type="text" id="user-name" />
  • form 요소는 label과 연결한다.
  • placeholder만으로 label을 대신하지 않는다.

07. 키보드(keyboard) 

/* 기본 포커스 스타일 제거 목적이 아니라,
   키보드 포커스에 명확한 스타일을 제공하기 위한 구성 */
a:focus-visible,
button:focus-visible,
input:focus-visible,
select:focus-visible,
textarea:focus-visible,
[tabindex]:focus-visible {
    outline: 2px solid #111;
    outline-offset: 2px;
}
  • 초점 표시를 제거하지 않는다.
  • 마우스 클릭 시 포커스 표시를 최소화해야 하는 경우 :focus-visible을 사용하고 키보드 탐색에는 명확한 초점 스타일을 제공한다.
  • 디자인상 outline을 사용할 수 없다면 배경, 테두리 또는 box-shadow로 현재 위치를 동일하게 구분할 수 있어야 한다.
  • 모든 기능은 키보드로 조작 가능해야 하며 Tab, Enter, Space, Esc로 메뉴, 버튼, 팝업을 사용할 수 있어야 한다.
  • 키보드 이벤트 및 Focus 제어는 개발 단계에서 구현 및 검수한다.
  • outline: none 또는 outline: 0만 사용해 키보드 포커스를 제거하지 않는다.

08. SVG와 아이콘 

<!-- 의미 없는 장식용 SVG -->
<svg aria-hidden="true" focusable="false" viewBox="0 0 24 24">...</svg>

<!-- 아이콘 단독 버튼 -->
<button type="button" aria-label="검색">
    <svg aria-hidden="true" focusable="false" viewBox="0 0 24 24">...</svg>
</button>

<!-- 의미를 전달하는 독립 SVG -->
<svg role="img" aria-labelledby="chart-title" viewBox="0 0 200 100">
    <title id="chart-title">월별 가입자 증가 그래프</title>
</svg>
  • 색상과 상태가 바뀌는 UI 아이콘은 inline SVG와 currentColor를 사용해 텍스트 색상 및 상태 스타일과 함께 제어한다.
  • 장식용 SVG는 aria-hidden="true"와 focusable="false"를 적용한다.
  • 아이콘 단독 버튼의 접근 가능한 이름은 SVG가 아니라 button의 aria-label 또는 숨김 텍스트로 제공한다.
  • 정보를 전달하는 독립 SVG는 role="img"와 고유한 title 연결을 제공한다.
  • CSS background-image: url(...svg)는 보조기기에 의미가 전달되지 않으므로 정보성 아이콘에 사용하지 않는다.
  • 외부 SVG URL을 img 또는 use로 불러오면 내부 path의 fill·stroke·상태를 CSS로 제어하기 어렵고, 경로·캐시·교차 출처 정책의 영향을 받을 수 있으므로 상태형 UI 아이콘에는 사용하지 않는다.
  • 고정된 로고나 콘텐츠 이미지는 <img src="logo.svg" alt="회사명">처럼 외부 SVG를 사용할 수 있다. 외부 SVG 자체를 금지하는 것이 아니라 용도에 맞게 구분한다.

09. 모션 축소

  • 화면 내 이동 범위가 크거나 반복·자동 재생되는 애니메이션이 포함된 컴포넌트는 prefers-reduced-motion을 지원하는 것을 권장한다.
  • 모션을 줄이더라도 콘텐츠, 상태 변화와 기능 결과는 동일하게 전달해야 한다.
  • 드로어·LNB·바텀시트·큰 레이어, 자동 재생 캐러셀·배너, 높이 변화가 큰 아코디언, 페이지 전환·패럴랙스·확대·스크롤 애니메이션, 반복되는 로딩·진행 상태에 우선 적용한다.
  • 큰 이동·확대·회전은 즉시 전환하거나 짧은 투명도 전환으로 대체하고, 자동 재생과 반복 모션은 정지한다.
  • hover 색상, focus 표시, checked 상태처럼 이동이 거의 없는 짧은 상태 전환에는 일괄적으로 적용하지 않아도 된다. 
/* 큰 이동이 있는 컴포넌트의 모션 축소 */
@media (prefers-reduced-motion: reduce) {
    .ui-bottom-sheet-panel,
    .ui-accordion-panel {
        animation: none;
        transition: none;
    }
}
// 자동 재생 Swiper를 사용하는 경우
const reduceMotion = window.matchMedia('(prefers-reduced-motion: reduce)').matches;

new Swiper('.js-banner-swiper', {
    speed: reduceMotion ? 0 : 300,
    autoplay: reduceMotion ? false : {
        delay: 3000
    }
});
  • progress는 최종 값을 즉시 표시하고, loading은 정적인 아이콘이나 상태 문구로 대체할 수 있다.
  • 아코디언과 바텀시트는 전환 시간을 제거해도 열린 내용, 닫힘 결과와 포커스 이동이 동일해야 한다.
  • Swiper 자동 재생을 사용하는 경우 정지·재생 버튼을 제공하고, 모션 축소 설정에서는 자동 재생을 시작하지 않는 방식을 권장한다.

10. 마크업 점검 사항

  • lang, title, main, heading 구조가 페이지의 실제 언어와 정보 순서를 설명하는지 확인한다.
  • 목록은 ul·ol, 표 데이터는 table·caption·th·scope, 용어와 설명 관계는 dl을 사용한다.
  • 클릭 동작을 div나 span으로 만들지 않고 이동은 a, 기능 실행은 button을 사용한다.
  • label, fieldset, legend로 입력 목적과 선택 항목의 그룹 관계를 제공한다.
  • 필수 입력, 오류, 완료 상태를 색상만으로 구분하지 않고 텍스트와 aria-describedby로 함께 전달한다.
  • display:none, hidden, aria-hidden 영역 안에 키보드 포커스 가능한 요소가 남지 않도록 한다.
  • 모달·바텀시트는 열린 뒤 제목 또는 첫 조작 요소로 포커스를 옮기고 닫은 뒤 실행 버튼으로 복귀시킨다.
  • 탭·아코디언 등 동적 UI는 aria-selected·aria-expanded·aria-controls와 실제 노출 상태를 일치시킨다.
  • 자동 재생, 깜빡임, 시간 제한 콘텐츠는 정지·연장 수단과 충분한 안내를 제공한다.
  • 200% 확대 및 모바일 가로·세로 화면에서 내용 잘림, 겹침, 가로 스크롤 여부를 확인한다.
  • 작업 완료 후 마우스를 사용하지 않고 Tab, Shift+Tab, Enter, Space, 방향키, Esc만으로 주요 기능을 점검한다.
